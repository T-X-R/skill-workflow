# 视频剪辑自动化平台开发思路

## 1. 项目目标

构建一个以 **Skill 为核心驱动** 的视频剪辑自动化平台。用户可以像搭积木一样，在网页端自由拖拽和组合不同 Skill 节点，形成属于自己的视频处理工作流。平台负责根据工作流顺序执行各个步骤，并允许用户在结果不理想时，通过自然语言提出修改要求，驱动后续调整。

这个平台的重点不是做一个固定流程的视频工具，而是做一个 **可编排、可扩展、可回改** 的自动化剪辑系统。

---

## 2. 核心理念

### 2.1 一切围绕 Skill
后端的处理逻辑尽量不写死在业务代码中，而是拆成一个个独立 Skill。每个 Skill 负责一个明确的剪辑步骤。

平台本身主要负责三件事：
- 管理有哪些 Skill 可用
- 让用户把 Skill 组合成工作流
- 按流程调度和执行 Skill

这样做的好处是，后续新增能力时，只需要新增 Skill，而不需要频繁修改平台主逻辑。

### 2.2 工作流优先
用户真正操作的对象不是“某个单点功能”，而是一整条视频处理流程。因此前端应该以 workflow 画布为中心，Skill 只是组成工作流的节点。

### 2.3 支持自然语言修正
视频生成结果很难一次完美，因此平台不能只支持“执行”，还要支持“修改”。当用户说“字幕太长了”“节奏不够快”时，系统应能够理解这是对当前流程中某一步或某几步的反馈，并引导后端进行调整。

---

## 3. 功能设计思路

### 3.1 可视化工作流编排
前端提供一个可视化工作流网站，用户可以：

- 从 Skill 列表中选择需要的节点
- 将节点拖到流程中
- 按自己的需求连接和排序
- 查看每个节点的大致功能说明
- 配置必要参数
- 保存和复用已有流程

每个 Skill 在界面中表现为一个气泡或节点，重点是让用户直观理解“这一步在做什么”。

### 3.2 Skill 执行调度
后端根据用户编排好的流程，逐步执行各 Skill。平台需要负责：

- 判断执行顺序
- 将前一步输出传给下一步
- 管理执行状态
- 记录日志和中间结果
- 在执行失败时返回清晰原因

这里建议平台做成一个通用执行器，Skill 只负责自己的具体处理逻辑。也就是说，平台做“编排与调度”，Skill 做“实际工作”。

### 3.3 缺失输入时向用户提问
有些 Skill 执行前，可能还缺少必要信息，例如：

- 视频字幕文件url
- 是否做口误剪辑
- 想要的时长

这时平台不应直接失败，而是应该中断在当前步骤，并提示用户补充信息。补充完成后，再继续从当前步骤往后执行。

这样可以让整个平台既自动化，又保留交互性。

### 3.4 自然语言修改结果
如果某一步做出来效果不好，用户不需要重新理解底层参数，而是可以直接说自然语言，比如：

- “字幕每行短一点”
- “配乐声音小一点”
- “视频光线亮一点”

系统的思路不是让用户直接操作底层算法，而是把自然语言理解为一种“修改意图”，再映射到后端执行逻辑中。常见方式包括：

- 修改某个 Skill 的传入参数
- 从某一步开始重新执行

这部分是平台体验的关键，也是后续最值得做深的部分。

---

## 5. 后端开发思路

### 5.1 平台后端只做调度，不做强耦合剪辑逻辑
后端应尽量做成“Skill 编排引擎”，而不是把所有视频逻辑写在主服务里。主服务负责：

- 渐进式披露加载 Skill
- 管理流程
- 执行流程
- 保存结果
- 响应用户修改

具体的视频处理动作由各 Skill 自己实现。这样后端会更清晰，也更方便扩展。

### 5.2 Skill 可以带脚本
Skill 不只是文字描述，它可以绑定自己的 Python 脚本，用于执行实际任务，例如调用：

- FFmpeg
- ASR 模型
- 字幕处理逻辑
- 视频分析模型
- 音乐处理工具

这样平台只需要定义一套 Skill 运行规范，就能逐步接入越来越多的视频能力。

### 5.3 执行过程可中断、可恢复、可重跑
因为视频处理通常耗时较长，而且中间步骤很多，所以后端需要支持：

- 执行中查看状态
- 失败后重试某一步
- 用户补充输入后继续执行
- 修改某一步后从该步骤往后重跑

不要把流程设计成“一次点运行就必须从头到尾完整成功”，那样用户体验会很差。

---

## 6. 自然语言修改能力的设计思路

这是整个平台区别于普通 workflow 工具的关键。

可以把自然语言修改理解为一个单独的“解释层”：

1. 用户描述不满意的地方
2. 系统判断问题更可能对应哪个 Skill 或哪个阶段
3. 系统生成建议动作
4. 用户确认后执行修改

例如：
- “字幕太长” → 优先考虑字幕切分相关 Skill
- “节奏慢” → 优先考虑镜头筛选、卡点、片段裁剪类 Skill
- “转场太突兀” → 优先考虑转场处理 Skill
